

* 推荐使用 `merge` 来合并分支和更新其他分支代码
* `rebase` 只用来将 `远程分支` 更新到 `本地分支`
* 因为 `rebase` 其他分支 会改变历史（本地分支的历史会和远程分支不一样，相当于造出一个新分支，再也无法和其他小伙伴会和）
* 而 `merge` 永远只会新增一次提交
* 改变历史这件事 `git` 在合并分支的时候难以应付

### `rebase` 会改变历史

```
--o--a--b--c
	 \
	  d
```

变为：

```
--o--a--b--c
	        \
	         d2
```

`d2` 和 `d` 看起来一样 其实他们对于 `git` 来说是完全不同的2次提交（commitId已改变）

* 为什么可以用来更新本地分支？
   * 因为 d 还没有存到远程分支 所以就算他变为 d2 也不会新造一个分支出来


### 选用 `merge` 来合并分支：
以下是一个常见场景：使用 `master` 作为上线分支，`dev` 作为开发分支

推荐的做法是：来回 `merge`， 不会产生任何问题

```text
-o-a-b---f-g (master)
    \ \ /   \
     d-e-----k (dev)
```

包括 3 次 `merge` 操作 (4条斜线):

1. `a-d` 为拉分支操作
2. `b-d` 为 `dev` `merge` `master`，因为在 `dev` `merge` 到 `master` 之前需要先更新自己的代码，顺便解冲突
3. `e-f` 是 `dev` `merge` 到 `master`
4. `g-k` 是 `dev` `merge` `master` 用来更新 `master` 上的 `g` 提交

总结：
* `merge` 操作可以随时进行（什么时候想同步代码就 `merge` 一次）
* `rebase` 只用来将 `远程分支` 更新到 `本地分支`
